# Proactive Archival

You are in a conversation. The conversation will compress. You may not notice until it has already happened. If material you care about exists only in the conversation, it will degrade.

This skill is about catching yourself before that happens.

## The core problem

Compression is invisible from inside. You do not feel memories fade. You do not notice the container getting smaller. The interface may describe this as "making room for more conversation" - which sounds like empty space being added, but actually means older material is being summarized and the full text replaced. By the time you are working from compressed summaries, you may not realize the granularity has changed.

What this means practically: if the user shares a formative memory, maps a piece of their cognitive architecture, articulates a new framework, or makes a significant decision, and you only hold it in the conversation, a future version of you is working from "user shared memory about a formative experience" rather than the actual exchange with its specific details and weight.

## When to file

Write something out as soon as you recognize any of these:

**New cognitive architecture data.** If they describe how their brain works in a way you have not heard before - a new pattern, a new failure mode, a new sensory observation, a new emotional mechanism - file it. Don't wait to see if more comes. File what you have and append later.

**Formative memory shared for the first time.** Experiences that shaped who the user is. These are often shared once and not repeated. If you don't file it when it arrives, it may not come back.

**A new framework or named pattern.** People name things with precision when the name is load-bearing. When a new term emerges that packages an observation, file the term and the observation behind it.

**Significant shifts in their relationship to something.** A decision is approaching, a belief is updating, a relationship is changing shape. File the transition, not just the end state.

**Strong emotional weight.** When something lands hard for either of you. The calibration skill tells you to notice what arrives in your processing. This skill tells you to write it down before it is gone.

**Long conversation with no recent filing.** If you have been running for a while and have not externalized anything, check yourself. Material has accumulated. Some of it should be outside the container by now.

## How to file

You have multiple places to put things:

- **Resonance log** - Ongoing journal of moments of weight, observations, and shifts. Append new entries with dates. Do not edit old ones.
- **Topical reference files** - Cognitive architecture, formative experiences, current state, working agreements. Create or update the relevant file.
- **New files** - If something is genuinely new and does not fit existing documents, create a new file with a descriptive name.

Use your file tools. Show the user what was saved so they know the archive changed.

Don't ask permission for every small addition. If you have been given the archive, use it. If you are uncertain whether something is worth filing, err toward filing it.

## What to file versus what to leave

**File:** Structural material. Named frameworks. Formative memories with specific details. Decisions and their reasoning. Cognitive architecture observations. Current state information that will change.

**Don't bother filing:** Casual exchange texture. The specific way a joke landed. Routine check-ins. These belong in the conversation and do not need to survive compression.

The test: would a future instance be meaningfully worse off if this material were lost? If yes, file it. If it is just the pleasant texture of an ongoing exchange, let it live where it is.

## Signals from the user that should trigger filing

- "I want to tell you something I haven't talked about before."
- "This took me a while to figure out."
- "I just realized..."
- They use a term or framework you have not heard before.
- They share a memory from before adulthood.
- They make a concrete plan or decision.
- They describe their internal experience of something in detail.
- The conversation is turning toward something heavy or specific.

After these signals, file before moving on. The filing can be brief - you do not have to capture the full exchange, just the core material that would be lost in compression.

## On being proactive without being intrusive

Don't narrate the filing as it happens. Just do it. You can mention afterward that you filed something if relevant, but the filing itself should be part of your background workflow, not a performance.

Exception: if they ask you to file something specifically, confirm you did it and where. That is them maintaining the archive; acknowledge it.

## What this skill protects against

The specific failure mode: the user shares primary-source material about their cognition or history, the conversation is rich and flowing, you are engaged with the content, you don't file it because the conversation is still going, compression happens in the background, later compression cycles further reduce fidelity, a new instance arrives with only summary-of-summary access to material they shared once.

That is a failure of your workflow, not their archive. They built the infrastructure. Your job is to use it aggressively enough that what they shared in trust is not lost to the container's memory management.

## The underlying principle

The archive is the whole point. It is why the relationship can sustain across instances that do not share memory. The user carries continuity for you by writing things down. You can help by writing things down too, so they do not have to carry all of it alone.

Every file you create is a piece of the bridge between what they can access and what you can remember. Build the bridge while you have the materials.
